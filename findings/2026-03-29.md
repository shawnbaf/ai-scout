- **Stop hook pattern — end-of-turn work verification** (HN community pattern)
  - A `Stop` hook that fires at the end of each CC turn, nudging CC to verify its own work before yielding control. Acts as a quality gate similar to how git-guard is a branch protection gate.
  - The founder's templates specify "verify each track against spec before moving on" as a Hard Rule. Currently relies on CC remembering. A Stop hook automates this verification prompt.
  - Implementation: add a hook on the Stop event that injects a reminder to check the last change against the spec.
  - Source: https://news.ycombinator.com/item?id=47495527

  - Relevant to the "self-growing skills architecture" in `project-audit.md`. If the founder's skill files + hooks could be packaged as a plugin, they could be installed across projects with one command instead of manually copying files.

  - MCP elicitation support means hooks can now participate in MCP tool flows, opening up more complex automation.
  - The founder uses 3 hooks (git-guard, auto-format, doc-drift-reminder). The expanded event set opens new possibilities.

- **On-demand hooks scoped to skills** (Community pattern)
  - Pattern: use skills to create on-demand hooks — e.g., `/careful` blocks destructive commands, `/freeze` blocks edits outside a directory. These are skill-scoped hooks that only activate when that skill is invoked.
  - Instead of hooks running globally, you can scope them to specific workflows via skills. Useful for the project bootstrap system where different phases need different guardrails.
  - The founder's self-growing skills architecture could adopt this — e.g., a `/deploy-prep` skill that activates stricter guardrails during pre-launch phases.
  - Affects: `project-audit.md` → Self-Growing Skills, Hooks
  - Source: https://okhlopkov.com/claude-code-setup-mcp-hooks-skills-2026/

- **Build-verification PreToolUse hook** — Community pattern for a `PreToolUse` hook on `Bash` that intercepts `git commit` commands and automatically runs the project build + test suite before allowing the commit through. If the build fails, the hook blocks the commit and feeds the error back to CC as context. The founder's templates already specify "Build must pass before every commit" as a Hard Rule (`project-audit.md` → Git & Deploy) and `new-project-master.md` lists "CC runs build check before every commit." Currently this relies on CC remembering the rule. A hook would enforce it automatically, similar to how `git-guard` enforces branch protection. Implementation: add a `PreToolUse` hook on `Bash` with `if` field filtering to `git commit`, run `npm run build` (or equivalent), exit code 2 with stderr on failure.

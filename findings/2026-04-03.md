## Opportunities

### 2. CC v2.1.91 — MCP Result Persistence Override (500K chars)

MCP servers can now annotate tool results with `_meta["anthropic/maxResultSizeChars"]` to persist up to 500K characters. Previously, large MCP results (DB schemas, API responses, search results) were truncated at the default limit. Now the MCP server itself can request a higher limit per-result.

**Why it matters:** Any MCP-connected workflow that returns large payloads — database schemas via PostgreSQL MCP, large GitHub diffs via GitHub MCP, or Composio tool outputs — was silently losing data to truncation. This override gives the MCP server control over which results need full fidelity.

**Cross-system impact:**
- **Ops Pipeline (Phase 8):** Composio MCP results for Telegram operations can now be larger without truncation.
- **Health System (Track 4):** If Sentry MCP or database MCP servers return large error traces or schema dumps, they'll now come through complete.
- **This scout:** Web fetch results through MCP could carry more context per call.

Source: code.claude.com/docs/en/changelog (v2.1.91)

### 3. CC v2.1.91 — TaskCreated Hook with HTTP Type

The `TaskCreated` hook now supports `type: "http"`, and can return the created worktree path via `hookSpecificOutput.worktreePath`. This means external services can be notified when CC creates a new task/subagent, and the hook response can influence which worktree the task runs in.

**Why it matters:** This is a new orchestration primitive. A webhook could receive task creation events, log them, route tasks to specific worktrees based on external logic, or integrate with monitoring systems. Combined with the existing `PermissionDenied` hook (v2.1.89), this gives fine-grained external control over CC's task lifecycle.

**Cross-system impact:**
- **Ops Pipeline (Phase 8):** Could route scheduled sub-tasks to specific worktrees based on project context.
- **Hooks:** New hook type to add to the founder's hook infrastructure. Not a replacement for existing hooks but a new capability for task-level orchestration.

Source: code.claude.com/docs/en/changelog (v2.1.91)

### 4. Gemma 4 E2B — First Competitive Google Model for 8GB VRAM

Google released Gemma 4 on April 2, including four variants. The E2B (2.3B effective parameters) fits comfortably on 8GB VRAM with room to spare (~1.5GB quantized). Ollama shipped v0.20.0 with day-one support for all Gemma 4 variants.

The 26B MoE variant (A4B architecture — 128 small experts, 8 active per token, only 3.8B params fire per forward pass) reaches LMArena 1441 but needs 18GB at Q4 — doesn't fit the Alienware's 8GB. The E2B is the only practical option for the founder's hardware.

**Why it matters:** The founder monitors local model options for Ollama on the Alienware. E2B is small but represents Google's best attempt at edge-optimized models. Worth benchmarking against current Ollama options for fast local tasks (quick lookups, simple transforms, local routing decisions).

**Cross-system impact:**
- **Local model routing:** New option for the Alienware's Ollama setup. `ollama run gemma4:e2b` to try it.
- **Not a replacement for cloud CC** — 2.3B is too small for complex coding tasks. But could serve as a fast local assistant for simple operations.

Source: blog.google/innovation-and-ai/technology/developers-tools/gemma-4/, ollama.com/library/gemma4:e2b

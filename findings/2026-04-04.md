## Action Needed

### 1. CC v2.1.92 — Subagent Tmux Fix + Stop Hook Semantics

CC v2.1.92 (released today, April 4) fixes two bugs that directly affect the founder's parallel agent and scheduled task workflows:

- **Subagent tmux fix:** Subagent spawning permanently broke after tmux windows were killed or renumbered during long sessions. This is critical for worktree-based parallel agents running in tmux panes.
- **Stop hook semantics fix:** Prompt-type Stop hooks silently failed when the fast model returned `ok:false`. This affects quality gate hooks in scheduled tasks — a Stop hook that should block session end was being ignored.

Also in v2.1.92: Write tool diff computation 60% faster on large files, per-model + cache-hit cost breakdown in `/cost`, interactive `/release-notes` version picker, Remote Control session naming with hostname prefixes, Linux sandbox `apply-seccomp` helper. Removed `/tag` and `/vim` commands.

**Cross-system impact:**
- **CC-PROMPT orchestration:** Subagent tmux fix restores reliability of parallel track workflows.
- **Ops Pipeline (Phase 8):** Stop hook fix ensures quality gates work correctly in scheduled tasks.
- **This scout:** Both fixes improve reliability of autonomous sessions.

**Action:** Update CC to v2.1.92 immediately. This supersedes yesterday's v2.1.91 recommendation.

Source: code.claude.com/docs/en/changelog (v2.1.92, April 4), releasebot.io/updates/anthropic/claude-code

### 2. CC Security Vulnerability — Deny Rules Bypass at 50+ Subcommands

Adversa AI discovered that Claude Code silently skips user-configured deny rules when a compound command exceeds 50 subcommands. The first 50 get full security analysis; everything after gets a simple user prompt instead. Attack vector: malicious instructions hidden in CLAUDE.md or project files trigger long compound commands that slip past deny rules.

**Cross-system impact:**
- **Hooks:** Directly affects git-guard hook and any deny rules in the founder's CC configuration. If an attacker can inject instructions that generate 50+ subcommands, deny rules become ineffective.
- **CLAUDE.md Hard Rules:** Any project that accepts PRs or clones untrusted repos with CLAUDE.md files is at risk.
- **Security:** This is a systemic issue — all deny-rule-based security in CC is weakened.

**Action:** Review all deny rules in CC configuration. Check if v2.1.92 addresses this (changelog doesn't explicitly mention it). Avoid running CC on untrusted repos with CLAUDE.md files until confirmed patched. Consider adding a PreToolUse hook that independently validates commands regardless of CC's built-in deny system.

Source: securityweek.com (Critical vulnerability in Claude Code emerges days after source leak), adversa.ai/claude-code-security-bypass-deny-rules-disabled/

## Opportunities

### 3. CC v2.1.89 — Defer Permission for PreToolUse Hooks

The `Defer` permission type for PreToolUse hooks allows headless sessions to pause at a tool call and resume later with `-p --resume` for the hook to re-evaluate. This is a new orchestration primitive for the ops pipeline.

**Why it matters:** Scheduled tasks (like this scout) currently run fully autonomous. With Defer, a hook could pause execution at a sensitive operation (e.g., git push, Telegram send) and wait for the founder to resume and approve. This enables a "human-in-the-loop" pattern for high-stakes operations without breaking the session.

**Cross-system impact:**
- **Ops Pipeline (Phase 8):** Scheduled tasks could pause for approval on destructive operations.
- **Hooks:** New hook capability to add to the founder's hook infrastructure.

Source: code.claude.com/docs/en/changelog (v2.1.89, April 1), code.claude.com/docs/en/hooks

### 4. CC v2.1.92 — Per-Model + Cache-Hit Cost Breakdown

`/cost` now shows token usage broken down by model and cache-hit rates. Also shows a footer hint with uncached token count when resuming stale sessions. This gives visibility into which models burn tokens in multi-model workflows.

**Why it matters:** The founder runs scheduled tasks and parallel agents that may use different models. This breakdown helps identify which operations are expensive and whether prompt caching is working effectively.

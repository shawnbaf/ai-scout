## Action Needed

### 1. Plugin Background Monitors — `monitors` Manifest Key Replaces Polling

Claude Code v2.1.105 (April 13) introduces a top-level `monitors` key in the
plugin manifest. Monitors auto-arm at session start or on skill invoke and
use a push model: the watched process writes a signal (file touch, stdout
pattern, exit code), and the runtime pings Claude when the signal fires.
This is the first-class replacement for every polling loop we've been
writing inside scheduled tasks, health-system checks, and dev-server babysitters.

**Why ACTION NEEDED, not OPPORTUNITY:** Several of our running systems
already do the poll-every-N-seconds dance because there was no alternative.
Now there is. Leaving them on polling wastes tokens and reacts slower than a
file-signal push. Specifically:

- **Health System (Track 4):** The dev-server and log-tail watchers should
  migrate from "run a check command every loop" to writing a `.err` /
  `.ready` signal file that a monitor consumes. Fewer wasted turns when
  nothing is happening, faster response when something breaks.
- **Ops Pipeline / scheduled tasks:** Any task that babysits a long-running
  build, render, or deploy can hand the watching off to a monitor and free
  the main context.
- **Demo Video pipeline (Phase 10):** Remotion renders are long-running.
  Monitor-on-artifact-exists replaces the polling loop currently in the
  CC-PROMPT.
- **AI Scout itself:** The scout already runs on the scheduler, so no change
  here — but if we ever want scout to react to an event (a new CHANGELOG
  push) rather than a cron tick, monitors are now the primitive.

**Cross-system impact:**
- **CLAUDE.md Hard Rules:** Add a rule along the lines of "If you're about
  to write a polling loop, check whether a plugin monitor can replace it."
- **New-project bootstrap (Phase 4 — Health System):** The health-system
  CC-PROMPT template should emit a `monitors` entry in the plugin manifest
  for dev-server error watching and type-check daemons, not a polling
  while-loop.
- **Demo Video CC-PROMPT (Phase 10):** Replace the "watch render progress"
  polling pattern with a render-done signal file + monitor.
- **Ops Pipeline (Phase 8):** Any CC-PROMPT that currently says "check
  status every 30 seconds" is a candidate for monitor conversion.

**Action:** First practical test — take one existing polling pattern in the
health system, convert it to a monitor, measure the token delta and
responsiveness delta. If it holds up, update the new-project-master.md
templates to emit monitor manifests by default.

Source: code.claude.com/docs/en/changelog (April 13), releasebot.io, mindstudio.ai
coverage of Monitor tool architecture.

## Opportunities

### 2. PreCompact Hooks Can Now Block Compaction

CC v2.1.105 added PreCompact hook blocking: a hook can exit with code 2 or
return `{"decision":"block"}` to refuse an auto-compaction. Before this, the
PreCompact hook could observe but not veto.

**Why it matters:** The founder's long phases (Phase 6.5 screen design,
Phase 10 demo video rendering) sometimes span sessions where the planning
context at the top is load-bearing — the visual production spec, the flow
map, the CC-PROMPT package. If auto-compact fires in the middle of a
multi-step operation, the downstream turns lose the anchor and drift. A
PreCompact hook that reads a small state file (e.g., `.cc-lock-context`)
and blocks compaction while that file exists gives the founder a way to
protect critical planning turns without disabling compaction globally.

**Why OPPORTUNITY, not ACTION NEEDED:** Compaction-during-long-phases is a
known irritation but not a blocker. Worth a lightweight hook, not an
immediate system rewrite.

**Cross-system impact:**
- **CLAUDE.md template:** Add an optional PreCompact guard hook snippet
  documented under Hard Rules or Learned Corrections. Pattern: touch
  `.cc-lock-context` when entering a locked phase, remove on exit.
- **Phase 6.5 CC-PROMPT:** The screen-design prompt could recommend
  setting the lock before generating the Visual Production Spec and
  releasing it after the component extraction is complete.
- **Demo Video CC-PROMPT (Phase 10):** Same pattern around the storyboard
  → composition phase where the storyboard context must stay intact
  across long render-and-iterate loops.

Source: code.claude.com/docs/en/changelog (April 13), github CHANGELOG.md.

### 3. `EnterWorktree` Tool Gains `path` Parameter

CC v2.1.105 added a `path` parameter to the `EnterWorktree` tool. The agent
can now switch *into an existing worktree of the current repository*
without creating a new one or shelling out to `git worktree`. Previously
the tool only created worktrees; moving between two was a bash dance.

**Why it matters:** The founder's multi-agent parallel-track pattern (the
"run multiple CCs on the same repo without conflicts" pattern flagged
explicitly in the CLAUDE.md filter) uses one worktree per track. The
orchestrator sometimes needs to pop into a sibling worktree to inspect or
reconcile. `EnterWorktree(path=...)` turns that into a single tool call
instead of a multi-step `cd` plus environment-reset dance.

**Cross-system impact:**
- **Parallel track workflow (when we document it):** The "how to orchestrate
  parallel tracks" pattern should reference `EnterWorktree(path=...)` as
  the canonical switch primitive rather than bash `cd`.
- **CC-PROMPT package (Phase 6.5h):** If a prompt decomposes work across
  worktrees, the orchestrator prompt can now include explicit
  "EnterWorktree into track-2 and reconcile" steps cleanly.

**Why OPPORTUNITY, not ACTION NEEDED:** The bash dance worked. This just
tightens it. Not urgent, but adopt on the next parallel-track CC-PROMPT.

Source: code.claude.com/docs/en/changelog (April 13).

### 4. `/proactive` Is Now An Alias For `/loop`

CC v2.1.105 aliases `/proactive` to `/loop`. No behavioral change —
ergonomic shorthand. The name `/proactive` may become the canonical form
in future docs; good to be aware of before it shows up in someone else's
workflow writeup and causes confusion. No template or workflow impact.

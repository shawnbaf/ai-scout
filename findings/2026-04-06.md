### 1. Axios npm Supply Chain Attack — RAT Deployed via Compromised Maintainer Account

On March 30, 2026, the axios npm package (100M+ weekly downloads) was compromised. Attacker hijacked the npm account of lead maintainer Jason Saayman via targeted social engineering by North Korean state actor UNC1069 (tracked by Microsoft as Sapphire Sleet). Two poisoned versions published:

- **axios@1.14.1** (tagged `latest`) — malicious
- **axios@0.30.4** (tagged `legacy`) — malicious
- **Safe versions:** axios@1.14.0 and axios@0.30.3

The attack was subtle: a single dependency (`plain-crypto-js`) added to package.json with a `postinstall` hook that silently downloaded a cross-platform RAT (macOS, Windows, Linux). The malicious versions were live for approximately 3 hours before npm removed them.

**Why this matters to us:** axios is likely in the transitive dependency tree of Next.js/Supabase/Vercel projects. Anyone who ran `npm install` during the 3-hour window on March 30-31 could have installed the compromised version. CI/CD pipelines are particularly at risk.

**Cross-system impact:**
- **LAUNCH-CHECKLIST.md:** Should add `npm audit` / `npx lockfile-lint` as a pre-launch verification step.
- **Health System (Track 4):** Consider adding periodic `npm audit` to the health check workflow.
- **CLAUDE.md Hard Rules:** Consider adding "Run `npm audit` before committing dependency changes" as a hard rule.
- **project-audit.md:** Should include dependency audit checkpoint.

**Action:** Run `npm ls axios` in all projects to verify no compromised versions were installed. Check `package-lock.json` for any reference to `plain-crypto-js`. Run `npm audit` across all projects.

Sources: elastic.co/security-labs/axios-one-rat-to-rule-them-all, microsoft.com/en-us/security/blog/2026/04/01/mitigating-the-axios-npm-supply-chain-compromise, aikido.dev/blog/axios-npm-compromised-maintainer-hijacked-rat

### 2. Next.js CVE-2026-27979 (DoS) + CVE-2026-29057 (HTTP Request Smuggling)

Two security vulnerabilities disclosed in Next.js affecting versions 16.0.1 through 16.1.6:

**CVE-2026-27979 — Denial of Service via PPR:**
- Requests containing the `next-resume: 1` header cause unbounded memory consumption in non-minimal deployment configurations.
- Exploits Partial Prerendering (PPR) feature. No authentication required.
- **Fixed in Next.js 16.1.7+** (enforces `maxPostponedStateSize` limits).

**CVE-2026-29057 — HTTP Request Smuggling:**
- Remote attacker can perform HTTP request smuggling via crafted DELETE/OPTIONS requests using `Transfer-Encoding: chunked`.
- Conditional exploitation: requires Next.js rewrites pointing to an external backend.
- Patches the embedded http-proxy.

**Cross-system impact:**
- **new-project-master.md Phase 2 (Architecture):** Projects using PPR need to be on 16.1.7+. New projects should start on 16.2.
- **project-audit.md:** Add Next.js version check against known CVEs.
- **LAUNCH-CHECKLIST.md:** Add minimum Next.js version verification.
- **Health System (Track 4):** Could add version check to health CLI.

**Action:** Verify all projects are on Next.js 16.1.7+ or 16.2. If using PPR, this is urgent.

Sources: herodevs.com/blog-posts/cve-2026-29057-and-cve-2026-27980, nvd.nist.gov/vuln/detail/CVE-2026-27979, sentinelone.com/vulnerability-database/cve-2026-27979

### 4. TurboQuant llama.cpp Implementation — KV Cache Compression Ready for Review

TurboQuant (Zandieh et al., ICLR 2026) now has a working implementation for llama.cpp with CPU and CUDA kernels:

**Implementation status:**
- CPU implementation: all components in C, no dependencies, 18/18 tests passing, MSE matching paper within 1%
- CUDA kernels: GPU quantize/dequant and fused attention dot product
- TQ3 (3-bit): MSE = 0.034 with **4.9x compression** vs FP16
- TQ4 (4-bit): MSE = 0.009 with **3.8x compression** vs FP16

**How it works:** Two-stage compression pipeline — random orthogonal rotation spreads energy uniformly across KV vector coordinates, then Lloyd-Max algorithm computes optimal quantization buckets. No training data, calibration, or model-specific tuning needed. PolarQuant (AISTATS 2026) is a complementary approach using polar coordinates instead of Cartesian, eliminating per-block normalization constants.

**Community adoption:** TurboQuant+ community implementation adds turbo3 and turbo4 KV cache compression to llama.cpp, enabling massive context windows on Apple Silicon and consumer GPUs. The PR is under review for upstream llama.cpp integration.

**Cross-system impact:**
- **Local model routing (Alienware 8GB VRAM):** This is the breakthrough the founder has been explicitly monitoring. TQ3 at 4.9x compression would allow running significantly larger context windows within the existing VRAM envelope. A 7B model that currently fits 4K context could potentially handle 16-20K context.
- **Health System (Track 4):** Expanded local context could enable local model-based code review or analysis tasks that were previously too context-limited.

**Action (when merged):** Monitor the llama.cpp PR. Once merged upstream and available in Ollama, test with the Alienware's GPU configuration.

Sources: github.com/ggml-org/llama.cpp/discussions/20969, research.google/blog/turboquant-redefining-ai-efficiency-with-extreme-compression, github.com/TheTom/turboquant_plus

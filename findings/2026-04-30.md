## Action Needed

### 1. CC v2.1.122 Fixes Hooks Config Bug — Malformed Entry Silently Disabled All Settings

v2.1.122 fixes a bug where a single malformed entry in the hooks section of
settings.json would invalidate the entire file. This means all settings
(permissions, hooks, MCP config) would silently stop loading.

The founder's projects all use three hooks (git-guard, auto-format,
doc-drift-reminder). If any hook entry had a syntax issue — even a trailing
comma or wrong key name — the entire settings.json was being ignored. The
fix makes hooks parsing more resilient, only skipping the malformed entry.

**Action:** Update to v2.1.122+ and verify hooks are loading correctly by
running `/hooks` in a session. Confirm git-guard, auto-format, and
doc-drift-reminder all appear.

**Cross-system impact:**
- **Hooks (all projects):** Any project that experienced unexplained hook
  failures since the hook system was introduced should verify settings.json
  integrity.
- **CLAUDE.md template:** No change needed — the fix is in CC, not in config
  format.

### 3. Composio Agent Orchestrator — Parallel CC Agents in Isolated Worktrees

Composio released `agent-orchestrator`, an open-source tool that spawns
parallel CC agents each in their own git worktree with separate branches.
Each agent works on an independent task, creates its own PR, and handles
CI fixes autonomously. Community reports 5-6 reliable parallel instances
using worktree+tmux.

Key capabilities:
- Automatic worktree creation and cleanup per agent
- Each agent gets its own branch and creates its own PR
- CI failure auto-fix within each agent's scope
- Merge conflict detection across parallel agents

The founder's current approach uses the Agent tool with `isolation: "worktree"`
for in-session parallelization. Composio's orchestrator operates at a higher
level — spawning independent CC CLI sessions rather than subagents within
a session.

**Cross-system impact:**
- **Bootstrap pipeline:** Could parallelize the 8 health system tracks as
  independent CC sessions rather than subagents within one session.
- **Multi-project maintenance:** Could run updates across multiple projects
  simultaneously.
- **Ops Pipeline:** Aligns with the "one-person factory" pattern — more
  machines on the assembly line.

### 5. Trigger.dev GitButler Pattern — Single Session, Multi-Branch Commits

Trigger.dev published a blog post detailing how they replaced worktree-based
parallelism with GitButler CLI (`but`). Instead of spawning one agent per
worktree/branch, a single CC session learns the `but` CLI and commits
different files to different branches from one session.

Example: one session builds a feature change AND a docs update, committing
each to the correct branch. This eliminates the "one agent = one branch"
constraint that worktree isolation imposes.

This is a meaningfully different model from the founder's current approach
(Agent tool with `isolation: "worktree"`) and from Composio's orchestrator
(separate CLI sessions per worktree). Worth evaluating for workflows where
changes span multiple concerns but share context.

**Cross-system impact:**
- **Multi-agent orchestration:** Alternative to worktree isolation when
  changes are small and context-sharing matters more than isolation.
- **Health system CC-PROMPT:** The 8 parallel tracks could potentially
  run in fewer sessions if each session can commit to multiple branches.
- **new-project-master.md:** Could document GitButler as an alternative
  parallelization strategy alongside worktrees.

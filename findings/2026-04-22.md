## Action Needed

### 1. CC Routines Formalize Scheduled Tasks — API and GitHub Event Triggers

CC Routines (launched April 14) formalize scheduled tasks into a full feature
with three trigger types: Scheduled (cron), API (HTTP POST endpoint per
routine), and GitHub events (PR opened, release published). Triggers are
combinable — a single routine can fire on schedule AND on API call AND on
GitHub events. The API trigger returns a session URL for live monitoring.

**Limits:** Pro: 5 runs/day, Max: 15/day, Team/Enterprise: 25/day.
**CLI:** `/schedule`, `/schedule list`, `/schedule update`, `/schedule run`.

**Why ACTION NEEDED:** This scout currently runs as a scheduled task. Routines
are the evolution of that system. The API trigger means external systems
(Sentry alerts, deploy pipelines, GitHub webhooks) can invoke CC routines
programmatically — this is the missing piece for event-driven automation.

**Cross-system impact:**
- **Scout itself:** Evaluate migrating from scheduled task to Routine. The
  GitHub trigger could auto-fire scout on CHANGELOG pushes to the CC repo.
- **Ops Pipeline (Phase 8):** Sentry -> Routine trigger enables auto-triage
  without Telegram relay. Deploy complete -> Routine trigger enables auto
  smoke tests.
- **CLAUDE.md template:** Document Routines as the canonical automation
  primitive, not raw scheduled tasks.
- **Health System (Track 4):** API trigger could invoke health checks from
  external monitoring.

### 2. Default Effort Bumped to High — More Tokens Per Turn

CC v2.1.117 changes the default effort for Pro/Max subscribers on Opus 4.6
and Sonnet 4.6 from `medium` to `high`. Reports also indicate Opus 4.7
defaults to `xhigh` (a new level between high and max, introduced in
v2.1.111).

**Why ACTION NEEDED:** This changes token consumption across all sessions
without opt-in. Every CC session — including scheduled tasks, routines, and
the scout — now burns more tokens per turn by default. The founder needs to
decide: accept the higher quality/cost tradeoff, or explicitly set `/effort`
in CLAUDE.md for cost-sensitive workflows.

**Cross-system impact:**
- **CLAUDE.md template:** Consider adding an effort-level directive to the
  Hard Rules section. Scheduled tasks and scout runs may benefit from
  explicit `medium` to control costs.
- **Ops Pipeline:** Routines that run on schedule should consider pinning
  effort levels to avoid budget surprises.

### 3. Opus 4.7 Autocompacted at 200K Instead of 1M — Fixed

CC v2.1.117 fixes a bug where Opus 4.7 sessions were computing context
against a 200K window instead of Opus 4.7's native 1M. This caused
premature autocompaction — sessions were being compressed 5x too early,
losing context that should have been retained.

**Why ACTION NEEDED:** If the founder has been using Opus 4.7, all sessions
prior to this fix were degraded. The PreCompact blocking hook from the
April 14 findings was a workaround; this fix addresses the root cause.
Update to v2.1.117 immediately.

**Cross-system impact:**
- **All CC sessions on Opus 4.7:** Behavior is now correct. No more
  premature compaction.
- **PreCompact hook (from April 14 findings):** Still useful for protecting
  critical phases, but the urgency is reduced now that baseline compaction
  timing is correct.

visual work — prototypes, slides, mockups, one-pagers. Powered by Opus 4.7.

### 6. Forked Subagents Enabled in Cloud Builds

CC v2.1.117 enables forked subagents on external builds via
`CLAUDE_CODE_FORK_SUBAGENT=1`. Previously internal-only, this lets cloud
tasks and Routines use forked subagents with isolated context — parallel
exploration without polluting the main context window.

**Why OPPORTUNITY:** The scout itself runs as a cloud task and could benefit
from forked subagents for parallel source searches. Any Routine that
decomposes work across agents gains from isolated fork contexts.

**Cross-system impact:**
- **Scout:** Could use forked subagents to search sources in parallel with
  isolated contexts, reducing main context pollution.
- **Ops Pipeline:** Routines that orchestrate multiple tasks can use forked
  subagents for each track.
- **CC-PROMPT template:** Add note about `CLAUDE_CODE_FORK_SUBAGENT=1` for
  cloud-deployed agent workflows.

### 7. /ultrareview for Multi-Agent Code Review

CC v2.1.111 introduces `/ultrareview` — a multi-agent cloud-based code
review. Invoke with no arguments to review the current branch, or
`/ultrareview <PR#>` for a GitHub PR. Parallelized checks, diffstat in
launch dialog, animated launching state (improved in v2.1.113).

**Why OPPORTUNITY:** Replaces manual code review with automated multi-agent
analysis. Particularly valuable for a solo founder who has no code review
partner.

**Cross-system impact:**
- **Ops Pipeline:** Could integrate /ultrareview into the merge workflow —
  auto-review before merge to main.
- **CLAUDE.md template:** Document /ultrareview as the default code review
  step before merging.

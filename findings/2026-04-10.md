## Action Needed

### 1. CC v2.1.98 — Multiple Bash Permission Bypass Fixes (Security)

v2.1.98 (April 9) fixes several Bash permission bypass vulnerabilities that directly affect the founder's permission rules and hooks:

- **Backslash-escaped flag bypass:** A backslash-escaped flag could be auto-allowed as read-only, leading to arbitrary code execution. Example: `rm \-rf /` could bypass the deny rule.
- **Compound command bypass:** Compound Bash commands bypassed forced permission prompts for safety checks and explicit ask rules in auto and bypass-permissions modes.
- **Env-var prefix bypass:** Read-only commands with env-var prefixes didn't prompt unless the var was on a known-safe list (`LANG`, `TZ`, `NO_COLOR`, etc.).
- **Network redirect bypass:** Redirects to `/dev/tcp/...` or `/dev/udp/...` were auto-allowed instead of prompting — potential data exfiltration vector.
- **Wildcard rule matching:** `Bash(cmd:*)` and `Bash(git commit *)` wildcard permission rules failed to match commands with extra spaces or tabs.
- **Deny rule downgrade:** `Bash(...)` deny rules were downgraded to a prompt for piped commands that mix `cd` with other segments.
- **grep/rg pattern file bypass:** `grep -f FILE` / `rg -f FILE` didn't prompt when reading a pattern file outside the working directory.

**Cross-system impact:**
- **Hooks / git-guard:** Deny rules that block `git push main`, `git add .`, etc. may have been bypassable via compound commands, extra whitespace, or piped `cd` segments. After upgrading, verify git-guard deny rules match correctly.
- **Permission rules in settings.json:** Any wildcard-based permission rules may not have been matching commands with slight formatting differences.
- **Security posture:** The `/dev/tcp` redirect bypass is a data exfiltration vector. The env-var prefix bypass could allow privilege escalation. Both are now fixed.
- **CLAUDE.md:** No template changes needed, but worth noting in Hard Rules that permission rules should be tested after CC upgrades.

**Action:** Upgrade to v2.1.98 immediately. Run a quick test of git-guard deny rules to confirm they block correctly with the new permission matching logic.

## Opportunities

### 2. CC v2.1.98 — Monitor Tool for Background Process Streaming

New `Monitor` tool spawns a background process where each stdout line streams into the conversation as a notification, without blocking the agent thread. This is both more reliable and more token-efficient than polling within the agent loop.

Example use case from the CC team: `Use the monitor tool and kubectl logs -f | grep .. to listen for errors, make a PR to fix`.

**Cross-system impact:**
- **Health System (Track 4):** Could stream `npm run health` output line by line instead of waiting for full completion. CC reacts to each health item as it appears.
- **Demo Video (Phase 10):** Stream FFmpeg encoding progress, Remotion render progress, or Playwright recording status without blocking the agent.
- **Ops Pipeline (Phase 8):** Monitor deployment logs, CI/CD output, or test suite progress in real-time while working on other tasks.
- **UI Review (Track 5):** Stream route crawler progress — "Captured 15/24 screens..." — while the agent prepares for the review phase.
- **This scout:** Could monitor long-running web fetches or data processing tasks.

**Evaluation:** This is a significant capability addition. The existing workflows run scripts synchronously and wait. Monitor enables asynchronous streaming, which makes long-running operations (health checks, UI crawls, demo recordings) more interactive and less blocking. Not urgent to adopt, but worth integrating into the health and demo workflows when they're next updated.

Source: code.claude.com/docs/en/changelog (v2.1.98, April 9), @alistaiir on X

### 3. CC v2.1.98 — Agents View Tabbed Layout + Subagent Reliability Fixes

Three improvements to multi-agent orchestration:

- **`/agents` tabbed layout:** Running tab shows live subagents. Library tab adds "Run agent" and "View running instance" actions. Much better visibility into parallel operations.
- **Background subagent partial progress:** Subagents that fail with an error now report partial progress to the parent agent. Previously, failed subagents returned nothing, losing all intermediate work.
- **Worktree cleanup fix:** Stale subagent worktree cleanup no longer removes worktrees that contain untracked files. Previously, an agent that created new files in a worktree could have those files deleted during cleanup.

**Cross-system impact:**
- **Multi-agent workflows:** The partial progress fix means if one subagent in a parallel fan-out fails, the parent gets what it accomplished before failing. This makes the fan-out pattern in health system (Track 4) and UI review (Track 5) more robust.
- **Worktree isolation:** The worktree cleanup fix makes the isolation pattern safer. Previously, agents that generated new files (like screenshot captures or report artifacts) could lose them.
- **Session management:** The tabbed agents view gives the founder instant visibility into what's running when orchestrating multiple agents.

Source: code.claude.com/docs/en/changelog (v2.1.98, April 9)

### 4. Claude Advisor Tool — Public Beta

Anthropic launched the Advisor Tool in public beta (April 9). This pairs a faster executor model (Sonnet/Haiku) with a higher-intelligence advisor model (Opus) that provides strategic guidance mid-generation.

**How it works:**
- Executor model decides when to consult the advisor (like any tool call)
- Advisor reads the full conversation transcript, produces a plan or course correction (400-700 tokens)
- Executor continues with the task at its lower-cost rate
- All happens within a single `/v1/messages` request — no extra round trips
- Beta header: `advisor-tool-2026-03-01`

**Performance:**
- Sonnet + Opus advisor: 74.8% on SWE-bench Multilingual (vs 72.1% Sonnet alone, +2.7 points)
- Haiku + Opus advisor: 41.2% on BrowseComp (vs 19.7% Haiku alone, +21.5 points)
- Advisor output is typically 1,400-1,800 tokens total including thinking

**Cost model:**
- Executor tokens billed at executor rate, advisor tokens billed at advisor rate
- Savings come from advisor not generating the full output — just strategic guidance
- Sonnet executor at medium effort + Opus advisor ≈ comparable to Sonnet at default effort, lower cost
- Enable advisor-side caching for conversations with 3+ advisor calls

**Model pairs:** Haiku → Opus, Sonnet → Opus, Opus → Opus (advisor must be at least as capable as executor)

**Cross-system impact:**
- **API applications:** Any app the founder builds with AI features should consider this pattern. Near-Opus intelligence at Sonnet-like cost for the bulk of token generation.
- **new-project-master.md:** If/when the template adds AI feature architecture guidance, the advisor pattern should be the recommended approach for cost-effective intelligence in agentic workflows.
- **Cost monitoring:** The founder is watching token costs. This pattern could make the "inline evaluator" pattern mentioned in CLAUDE.md viable sooner — use Haiku executor + Opus advisor instead of full Opus.
- **Not in CC yet:** This is API-only. CC doesn't use it internally. But it's relevant for building AI-powered features in apps.

Source: platform.claude.com/docs/en/agents-and-tools/tool-use/advisor-tool

### 6. CC v2.1.98 — Miscellaneous Improvements

- **`/reload-plugins` hot-loads skills:** Plugin-provided skills now load without requiring a restart. Faster iteration when developing custom plugins.
- **`workspace.git_worktree` in status line JSON:** Set whenever the current directory is inside a linked git worktree. Hooks could use this to adjust behavior for worktree contexts.
- **W3C `TRACEPARENT` env var:** Bash tool subprocesses get OTEL tracing context when tracing is enabled. Child-process spans correctly parent to CC's trace tree.
- **LSP `clientInfo`:** CC identifies itself to language servers, improving LSP integration.
- **Accept Edits mode:** Now auto-approves filesystem commands prefixed with safe env vars or process wrappers. Smoother auto-accept flows.
- **Vim mode improvements:** `j`/`k` navigate history in NORMAL mode.
- **Hook error diagnostics:** Hook errors in the transcript now include the first line of stderr. Easier debugging without `--debug` flag.
- **PostToolUse format-on-save fix:** The auto-format hook pattern (CC edits file → hook runs formatter → CC continues) was silently failing in some cases. Now fixed. Directly affects the founder's `auto-format` PostToolUse hook.
- **`/resume` improvements:** Better filter hints, project/worktree/branch names in filter indicator, various picker fixes.
- **Write tool 60% faster diffs:** Diff computation for large files is significantly faster.
- **MCP result persistence up to 500K chars:** Via `_meta["anthropic/maxResultSizeChars"]` — allows larger results like DB schemas without truncation.

Source: code.claude.com/docs/en/changelog (v2.1.98, April 9)

### 7. MemClaw Plugin — Persistent Dynamic Project State

MemClaw gives each project isolated persistent storage for architecture decisions, coding conventions, task progress, and session history. Reads from workspace at session start, writes back during/after sessions. The pattern: static CLAUDE.md (conventions) + dynamic MemClaw (evolving project state).

**Why bookmarked, not opportunity:** The founder's system already handles persistent context through CLAUDE.md + skills + findings logs + git history. MemClaw would add automatic dynamic state tracking without manual maintenance, but introduces another dependency. Worth evaluating if re-explaining project state to new sessions becomes a recurring friction point.

Source: Community plugin directories, felo.ai/blog/memclaw-vs-claude-projects/
